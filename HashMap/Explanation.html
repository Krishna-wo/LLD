01 · Foundation

The overall shape

map is a plain Java array of Node references — separate chaining. What makes this implementation different from a textbook version is that every bucket is not just a linked list — it's a doubly linked list with two permanent sentinel nodes: a head (key = null) and a tail (key = null) that never move and never hold real data. Real entries are always inserted between the head and whatever came after it.

map.length = 4 · each bucket = HEAD ⇄ [real nodes...] ⇄ TAIL · double-headed arrows = prev/next links
Why sentinels?

Using dummy head/tail nodes means put and remove never have to special-case "is this the first node in the bucket?" or "is this the last one?" — head.next and tail.prev always exist, so every insert/delete is a uniform 4-pointer rewire. This is the classic sentinel-node trick from doubly linked list design.

02 · Setup

Constructor — wiring up empty buckets

CustomHashMap(){ map = new Node[INITIAL_SIZE]; // array of 4 nulls for(int i=0; i<INITIAL_SIZE; i++){ map[i] = new Node<>(null, null); // this bucket's HEAD sentinel map[i].next = new Node<>(null, null); // this bucket's TAIL sentinel map[i].next.prev = map[i]; // link them back to each other } }

For every index 0..3, two throwaway Node objects are created and wired together: HEAD.next = TAIL and TAIL.prev = HEAD. Nothing else points at HEAD or TAIL from outside — they exist purely as anchors.

Repeated identically for map[0], map[1], map[2], map[3]
03 · Insertion

put(key, value) — trace with real collisions

We'll insert "name", "age", "city", then "id" — using real Java String.hashCode() values so the buckets below are accurate, not illustrative:

All four collide into bucket 3 — a genuine worst-case chain, which is exactly what makes this a good example of the linked-list-per-bucket design.

public void put(K key, V value){ Node node = findNode(key); if(node != null){ node.val = value; return; } // key exists → overwrite int bucket = Math.abs(key.hashCode()) % map.length; Node head = map[bucket]; Node newNode = new Node(key, value); Node old_first = head.next; // whoever was right after HEAD head.next = newNode; newNode.prev = head; newNode.next = old_first; old_first.prev = newNode; countOfNodes++; if(countOfNodes > LOAD_FACTOR * map.length) rehash(map.length * 2); }

New nodes are always spliced in right after HEAD — this is a LIFO / most-recently-inserted-first ordering within a bucket, not append-to-end.

Step 1 — put("name", "Jo") into empty bucket 3

bucket[3]: HEAD ⇄ "name" ⇄ TAIL

Step 2 — put("age", 30): bucket 3 collides, insert at head

bucket[3]: HEAD ⇄ "age" ⇄ "name" ⇄ TAIL — newest is always closest to HEAD

Step 3 — put("city", "LA"): three real nodes now chained

bucket[3]: HEAD ⇄ "city" ⇄ "age" ⇄ "name" ⇄ TAIL
Then put("id", 7) hits the same bucket

That's the 4th node overall. countOfNodes becomes 4, and 4 > LOAD_FACTOR * map.length → 4 > 0.75 × 4 = 3 is true. This single insert both extends the chain to 4 nodes and immediately fires rehash(8) before put returns. We'll pick this up in the rehash section below.

04 · Lookup

findNode() and get()

public Node findNode(K key){ int bucket = Math.abs(key.hashCode()) % map.length; Node head = map[bucket]; while(head != null){ if(head.key != null && head.key.equals(key)) return head; head = head.next; } return null; }

This walks forward from the HEAD sentinel through every node — including eventually the TAIL sentinel — comparing keys. Because head.key != null guards the sentinels, they're silently skipped rather than matched.

findNode("name") — linear scan stops the moment key.equals(key) matches

get(key) is a one-line wrapper: call findNode, return node.val if found, else null. Worst case for a lookup is O(chain length) — which is exactly why rehash exists: keep chains short as the map grows.

05 · Deletion

remove(key) — 4-pointer splice

public void remove(K key){ Node nodeToRemove = findNode(key); if(nodeToRemove == null) return; Node prevNode = nodeToRemove.prev; Node nextNode = nodeToRemove.next; prevNode.next = nextNode; nextNode.prev = prevNode; countOfNodes--; }

Removing "age" from the 3-node chain above — its neighbors just reach around it:

prevNode.next skips forward, nextNode.prev skips backward — "age" drops out of both directions
06 · Resizing

rehash() — the full trace that put("id", 7) triggers

Continuing the exact example from Section 3: bucket 3 holds HEAD ⇄ "city" ⇄ "age" ⇄ "name" ⇄ TAIL and we now insert "id", which also hashes to bucket 3.

countOfNodes crosses the threshold

"id" is inserted the normal way first (bucket 3 becomes a 4-chain), then countOfNodes becomes 4. Check: 4 > 0.75 × 4 (=3) → true. rehash(map.length * 2) → rehash(8) is called.

A brand new array is allocated

Node[] newMap = new Node[8], and all 8 buckets get fresh HEAD/TAIL sentinel pairs — identical setup to the constructor, just at the new size.

Every old bucket is walked

The outer loop for(Node curr : map) visits each of the 4 old bucket HEAD sentinels in turn. The inner while(curr != null) walks that bucket's whole chain — sentinels included.

Sentinels are skipped, not recreated

if(curr.key == null){ curr = curr.next; continue; } — the old HEAD/TAIL objects are simply stepped over. Only real key/value nodes get relinked.

Each real node is re-bucketed and spliced into the new array

For each real node, bucket = Math.abs(curr.key.hashCode()) % newSize is recomputed against the new size (8, not 4) — so nodes can land in a completely different bucket than before. It's then linked in right after the new bucket's HEAD, same LIFO-style insert as put uses.

This is the payoff of rehashing: three of the four keys stay together in the new bucket 3, but "age" moves out to bucket 7 — the 4-long chain splits into a 3-chain and a 1-chain, which is exactly the point of growing the table.

Iteration order inside rehash's while loop

One subtlety worth tracing carefully: nextNode = curr.next is saved before curr gets relinked into the new map. This is essential — once curr.next is overwritten to point into newMap, the old chain would otherwise be lost mid-walk.

while(curr != null){ if(curr.key == null){ curr = curr.next; continue; } int bucket = Math.abs(curr.key.hashCode()) % newSize; Node newHead = newMap[bucket]; Node oldFirst = newHead.next; Node nextNode = curr.next; // ← saved BEFORE curr.next is reassigned newHead.next = curr; curr.prev = newHead; curr.next = oldFirst; oldFirst.prev = curr; curr = nextNode; // ← walk continues using the saved pointer }
Why the final chain order flips

The old bucket 3 chain was city → age → name → id (id inserted most recently, closest to old HEAD). Walking it left-to-right and always inserting each node "closest to new HEAD" means the last one walked ends up first in the new bucket — hence new bucket 3 reads id → name → city, the reverse of the old bucket's real-node order.

07 · Review

A few things worth double-checking in this code

These aren't part of "how it works" — they're spots where the current code may not do what you intend. Flagging them since you're clearly iterating on this as a learning project.

Raw types everywhere

Node is used unparameterized in most methods (Node node = findNode(key), Node[] map, Node head = map[bucket]) instead of Node<K,V>. This compiles with unchecked-warning noise and throws away compile-time type safety — the compiler can no longer catch a String being put where an Integer was expected, for instance.

get()'s unchecked cast

return node == null ? null : (V)node.val; — since node is a raw Node, node.val is typed as Object, so this cast is unchecked and will compile with a warning. It'll work at runtime here since val really was stored as V, but the compiler can't verify it.

TAIL is included in the linear scan

findNode's while(head != null) will walk as far as the TAIL sentinel (key is null, so it's skipped by the guard) and then stop only because TAIL.next is Java's default null for an unassigned field. It works, but it's relying on TAIL.next never being explicitly set rather than an explicit "stop at tail" check — worth a comment so a future reader doesn't "fix" it by pointing TAIL.next somewhere.

@Setter on Node is unused, and slightly risky

Lombok's @Getter @Setter generates public setters for key, val, next, prev — but the class also has hand-written field access everywhere (node.val = value works either way since these are package-private fields accessed from the same file). The setters aren't harmful, just redundant here, and they do make it possible for outside code to reach in and corrupt the linked-list pointers directly.

No null-key handling

put, get, remove, and findNode all call key.hashCode() or key.equals(...) directly. A null key passed by a caller throws NullPointerException rather than being handled — which may be fine depending on your requirements, but is worth being a deliberate choice rather than an accident, especially since null is already the sentinel marker used internally for HEAD/TAIL.
